## What's changed

### Multi-Layer Intelligence Stack for comprehensive codebase understanding

OpenKit 0.8.0 introduces a 4-layer intelligence stack that enables comprehensive codebase understanding, ensuring agents read codebases **broadly** (find all relevant context), **deeply** (understand how code works), and **reliably** (never miss critical context).

**Layer 1 — Structural Intelligence:**
- Enhanced project graph with type flow tracking (assignment, parameter, return, property)
- Lexical scope context tracking with variable bindings
- Extended schema: `type_flows` and `scope_contexts` tables
- Query capabilities: trace type flows, scope chains, decorator searches
- Extended `nodes` table: module_type, package_name, is_test, is_config
- Extended `symbols` table: return_type, params_json, decorators_json, parent_symbol_id, scope_chain, start/end columns

**Layer 2 — Semantic Intelligence:**
- Pattern recognition: api-usage, validation, error-handling, architectural patterns
- Data flow analysis with BFS traversal for tracing values through transformations
- Usage pattern mining for actual code usage fingerprints
- Multi-source semantic search combining embeddings + patterns + usage + graph
- `code_patterns` table with FTS index for fast pattern search
- Integration with embedding indexer for enhanced chunk metadata

**Layer 3 — Intent Intelligence:**
- LLM-augmented business logic extraction
- Extractors: business rules, edge cases, design patterns, constraints, data transformations
- SHA256-based intent caching with code-change invalidation
- Confidence scoring via cross-validation with structural data
- `code_intents` table for cached extractions
- Background extraction triggers on file save

**Layer 4 — Context Assembly:**
- Smart orchestration: parallel queries across L1 + L2 + L3
- Budget management: 40% critical, 30% important, 20% supplementary, 10% buffer
- Multi-layer result ranking combining structural, semantic, and intent signals
- Session memory maintaining working set across tasks
- Three modes: task (focused), session (working set), project (broad exploration)

**Tools:**
- `tool.comprehensive-context` — main context gathering (task/session/project modes)
- `tool.data-flow-trace` — trace data flows through code
- `tool.type-flow-trace` — trace type flows
- `tool.pattern-search` — search by code patterns
- `tool.business-rule-query` — query business rules
- `tool.constraint-query` — query constraints

**Configuration:**
- `.opencode/openkit.runtime.jsonc` → `codeIntelligence` section
- Configurable per-layer enable/disable, LLM model selection, embedding integration
- See `docs/configuration/code-intelligence.md` for full reference

## Notes

**Performance expectations:**
- Initial indexing: 1-3 minutes for medium codebases (varies by size and complexity)
- Storage overhead: 2-3x graph DB size (depends on pattern/intent density)
- Query latency: 2-5 seconds for comprehensive context assembly (task mode fastest, project mode slowest)

**Migration from 0.7.0:**
- No breaking changes to existing workflow or runtime APIs
- New schema migrations run automatically on first index
- Existing project graphs are extended in-place (non-destructive)
- L2/L3 layers are opt-in via runtime config (L1 structural always enabled)
- Embedding integration is backward compatible (existing embeddings preserved)

**Audit findings resolved:**
- Finding 1 (Critical): Version metadata drift — fixed with sync:version script and hardened updateVersionMetadata()
- Finding 2 (Important): verify:all coverage gap — fixed by adding release tests to verify:all
- Finding 3 (Medium): workflow-kernel error handling — fixed with structured error results

**Configuration defaults:**
- L1 (Structural): always enabled
- L2 (Semantic): patterns enabled by default, data flow on-demand
- L3 (Intent): disabled by default (LLM-bound, enable for deeper analysis)
- L4 (Context Assembly): enabled when project graph is available

**Backward compatibility:**
- All existing runtime, workflow, and tool APIs remain unchanged
- New intelligence layers are additive (zero impact if not configured)
- Existing tests continue to pass without modification
