## What's changed

### Multi-session workflow isolation (one-shot cutover)

- introduce per-session OpenKit sessions: each `openkit run` tab gets its
  own `s_<id>` and its own per-session workflow-state mirror, so two
  tabs in the same repo can hold two different work items at the same
  time without colliding on `active_work_item_id`
- migrate `work-items/index.json` to schema v3
  (`openkit/work-items-index@3`) — adds per-item `current_session_id`
  and explicit `lane` / `status` fields, drops the global
  `active_work_item_id`
- rotate the top-level `.opencode/workflow-state.json` to a tiny
  forwarding stub (`openkit/legacy-stub@1`); legacy snapshots are kept
  under `workflow-state.json.legacy.*` with a 10-file rotation cap
- add session lifecycle commands: `openkit sessions list|show|resume|
  abandon|kill|downgrade-index`, `openkit dashboard`, and
  `openkit finish` (with `/finish` slash equivalent)
- add session-start banner and statusline tag scoped by
  `OPENKIT_SESSION_ID`
- bind slash lanes to a session: `/quick-task`, `/delivery`, `/migrate`
  refuse to repurpose a session that is already bound to a work item
- heartbeat + orphan scanner: a session that stops writing
  `heartbeat.json` for more than 10 minutes is auto-marked `orphan` and
  resumable from any tab
- five new `openkit doctor` checks: `sessions-index-readable`,
  `legacy-mirror-rotation`, `orphan-sessions-count`,
  `worktree-orphan-mismatch`, `pid-cleanup`

### Breaking changes

- **Schema v2 → v3 migration is required and runs on first
  `openkit run` after upgrade.** It is idempotent. It rewrites
  `.opencode/work-items/index.json` and rotates the top-level
  `workflow-state.json`. Multi-machine setups should not run two
  v0.7.0 boots concurrently against the same `.opencode/` until the
  first one completes.
- **`active_work_item_id` is removed from the v3 index.** Any custom
  tooling that read `active_work_item_id` directly must switch to the
  resolver: read `OPENKIT_SESSION_ID`, then look up the per-session
  mirror via `OPENKIT_WORKFLOW_STATE`, or call
  `resolveSession({ env, repoRoot }).workItemId` from
  `src/runtime/sessions/session-resolver.js`.
- **Top-level `.opencode/workflow-state.json` no longer represents
  live workflow state.** It is now a forwarding stub. Tools must read
  the per-session mirror at `OPENKIT_WORKFLOW_STATE`.
- Any in-flight v2 work item that the migrator could not bind to a
  live process becomes a **synthetic orphan session**
  (`s_orphan_<8hex>`). Visible via `openkit sessions list` and
  `openkit dashboard`. Operators choose `resume` or `abandon` per item.
- New runtime dependency: `proper-lockfile` (used by
  `sessions/index.json` and `work-items/index.json` writers).

### Migration guide

1. Commit or stash any in-flight changes in your repo before
   upgrading.
2. `openkit upgrade` (or `npm install -g @duypham93/openkit@0.7.0`).
3. Run `openkit run` once. The startup migrator will:
   - rewrite `.opencode/work-items/index.json` to v3 (idempotent),
   - rotate `.opencode/workflow-state.json` into
     `workflow-state.json.legacy.<timestamp>`,
   - emit one synthetic orphan session per in-flight v2 work item
     that has a managed worktree.
4. Run `openkit dashboard`. For each synthetic orphan, choose:
   - `openkit sessions resume <id>` to attach a fresh tab and continue,
   - or `openkit sessions abandon <id>` to drop it (worktree retained
     unless you pass `--remove-worktree`).
5. Run `openkit doctor` and confirm all five session checks return
   `ok`.
6. **Rollback (incident only).**
   `openkit sessions downgrade-index` rewrites the v3 index back to v2
   and restores the most recent `.legacy.*` mirror. Per-session state
   is not representable in v2 and is lost when downgrading; only run
   this if you must roll back the binary.

### Notes

- macOS sleep / system suspend longer than 10 minutes will mark
  active sessions as orphan. Resume is one command. Documented in the
  operator runbook.
- Worktree paths with whitespace or symlinks are unchanged; the
  worktree manager already handled them.
- `/finish` is gate-bound: it will not finish until the lane's
  approval gate (`quick_verified`, `qa_to_done`, or
  `migration_verified`) has been approved.
