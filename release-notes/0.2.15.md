## What's changed

- Add conditional parallel execution semantics for `Full Delivery` and `Migration` instead of treating parallelism as an automatic lane behavior.
- Introduce explicit `parallelization` workflow-state metadata so plans can bless `none`, `limited`, or `enabled` execution modes.
- Harden full-delivery execution with:
  - task `concurrency_class` support
  - task allocation validation
  - integration readiness checks
- Add migration-specific slice board support instead of reusing the full-delivery task board model:
  - migration slice validation
  - migration slice ownership
  - parity-oriented slice status flow
- Update canonical workflow docs and plan templates so PM, BA, Architect, and Tech Lead must assess whether parallel execution is actually safe before worker pools are activated.
- Add Tech Lead and maintainer guidance for the new conditional parallel model:
  - `docs/operations/runbooks/tech-lead-task-decomposition.md`
  - `docs/maintainer/conditional-parallel-execution-note.md`

## Notes

- This release does not make parallel execution automatic; it makes it explicit, plan-driven, and bounded.
- Quick work remains non-parallel, and migration slices remain distinct from the full-delivery task board so lane semantics stay intact.
