## What's changed

- Restored OpenCode slash-command discovery (`/delivery`, `/quick-task`, `/migrate`, `/finish`, `/handoff`, `/start-work`, `/write-solution`, `/execute-solution`, `/switch-profiles`, `/switch`, `/configure-agent-models`) after the v0.9.0 source-tree reorganization moved files under `src/` but left the materialize step copying them to `<kitRoot>/src/...` instead of the `<kitRoot>/{commands,agents,skills,context}/` paths OpenCode auto-discovers by convention.
- Added `stageOpenCodeDiscoveryLayer`, `validateMaterializedKitLayout`, and a fail-closed `MaterializationError` so any future bundle/layout regression rolls back the install with an actionable hint.
- Added `detectKitLayoutDrift` + `repairKitLayout` so existing v0.9.0/.1/.2 installs auto-heal on the next `openkit run` — no manual `openkit upgrade` required. A new `repaired-layout` action surfaces `"OpenKit detected missing OpenCode discovery layer and repaired it automatically. Layout: 11 commands, 7 agents, 20 skills."` so operators see what happened.
- Expanded `OPENKIT_OPENCODE_BUNDLED_ASSETS` with `/finish`, `/handoff`, `/start-work` (public per CLAUDE.md/AGENTS.md); classified `browser-verify`, `configure-embedding`, `init-deep`, `refactor`, `stop-continuation` as `internalOnly` with explicit rationale.
- Normalized post-reorg path references in `src/hooks/session-start.js` and `src/openkit-runtime/lib/workflow-state-controller.js` (`getRuntimeStatus`, `runDoctor`, `getRegistry`, `getInstallManifest`, `getContractConsistencyReport`) plus the `resolveKitRoot` probe to Layer B (`<kitRoot>/src/...`) consistently.
- Aligned `install-manifest.json` `repositoryInternalRuntime`, `wrapperFacingMetadata`, and `migration.currentRuntimeManifest` with the post-reorg layout (was masked by a stale assertion in `registry-metadata.test.js`).
- Wired `sourceCommandsMissingFromBundle` into the `verify-install-bundle` drift check so `npm run verify:install-bundle` exits non-zero when a new `src/commands/*.md` is added without classification.
- Added regression tests: `materialize-global` (4 integration cases), `stage-discovery-layer` (3 unit), `validate-materialized-kit-layout` (5 unit), `drift-detector` (3 unit), `asset-manifest-coverage` (invariant).
- Excluded `src/.opencode/` from the npm package `files` to keep local dev workflow-state artifacts out of published tarballs.

## User action

Existing v0.9.0/.1/.2 users should upgrade:

```bash
npm install -g @duypham93/openkit@latest
openkit run    # auto-detects Layer A drift and repairs in place
```

`openkit upgrade` (full re-materialize) also works and is the safer choice for installs that have other reported drift.

## Notes

- Backward compatible: no lane names, slash commands, or workflow-state fields changed. Existing project shims under `<projectRoot>/.opencode/` are untouched.
- Architectural change: install layout now follows a 3-layer model — Source (`src/...` in repo) → Kit-root Layer A (`<kitRoot>/{commands,agents,skills,context}/` for OpenCode discovery) + Layer B (`<kitRoot>/src/...` for kit-internal modules) → Project shim (`<projectRoot>/.opencode/openkit/` unchanged). See `docs/superpowers/specs/2026-05-16-openkit-discovery-layer-fix-design.md` for the full design.
- Deferred items tracked in `docs/superpowers/plans/2026-05-16-discovery-layer-pr2-p1-cleanup.md` and `pr3-p2-followup.md`: `builtin-commands.js` path normalization, doctor Layer A checks, fixture drift in `materialize.test.js`/`merge-policy.test.js`/`doctor.test.js`, `verify:discovery-layer` aggregate script, `MaterializationError` consumer wiring in non-`run.js` CLI commands, commands-vs-skills classification shape unification.
